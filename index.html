Tool 02

Values, Observable, Teachable (VOT)

For each of the four core values — Humble, Hungry, Happy, Honest — here are the observable behaviors we can hire by, coach to, and reward. Use this in every 1-on-1 and quarterly review.

How to Use the VOT

In every quarterly 1-on-1 and annual review, score each staff member on all four values using a 1–5 scale. A "Right Person" lives these values naturally. A "Right Seat" question is separate — someone can be a great fit to our culture but wrong for the role. Both matter.

Humble

Humility is the foundation. It's not thinking less of yourself — it's thinking of yourself less. We take Jesus seriously and not ourselves.

  • Gives credit freely; takes blame personally
  • Asks for help without ego; actively seeks feedback
  • Serves outside their job description without complaint
  • Deflects praise toward the team and toward God
  • Listens more than they talk; curious, not defensive
  • Doesn't hoard information, decisions, or control

Hungry

Hungry people are self-motivated. They don't wait to be told. They pursue excellence in ministry because they love Jesus and love the mission.

  • Pursues personal spiritual formation proactively
  • Reads, learns, and brings new ideas without being asked
  • Sets goals beyond what's required; moves the ball
  • Champions improvement and resists "we've always done it this way"
  • Takes on extra responsibility when the mission requires it
  • Execution-oriented — follows through without reminders

Happy

Happy doesn't mean always smiling. It means we know who we are in Christ, why we're here, and we find genuine joy in the work — even the hard parts.

  • Brings energy to the team, not drama
  • Practices personal disciplines that sustain health (Sabbath, prayer, margin)
  • Celebrates wins — big and small — genuinely and publicly
  • Treats volunteers and guests with consistent warmth
  • Doesn't complain without offering a solution
  • Can receive hard feedback or news without falling apart

Honest

We say hard things with grace. No triangulating, no gossip, no sweeping things under the rug. Authenticity is not optional in this culture.

  • Says what they mean in the room, not after the meeting
  • Admits mistakes quickly and owns outcomes fully
  • Brings problems forward — not sideways or backwards
  • Addresses conflict directly with the person involved
  • Keeps confidences; never leaks private matters
  • Doesn't shade truth to protect comfort or avoid awkwardness

Tool 05

Quarterly Rocks

The 3–7 most important things to accomplish this quarter. If everything is a priority, nothing is. Every rock is specific, measurable, and done in 90 days.

The Rock Formula

"By end of Q2 2026, [this specific thing] will be done and we'll know it because [here's the measurable proof]." Rocks are reported On Track / Off Track every L10. Red rocks go directly to Issues.

Company Rocks — Q2 2026
Individual Rocks — Q2 2026
Rock Review Cadence Weekly L10
Every person reports every rock every week — On Track or Off Track only
  • On Track (Green) — No discussion needed. Move on immediately.
  • Off Track (Red) — Brief update only. Moves to Issues List for IDS if needed.
  • Completed — Celebrate briefly. Confirm done with clear evidence.
  • Rocks are never moved or changed mid-quarter except genuine emergency (extremely rare).
  • End of quarter: hold a Quarterly Business Owner's meeting (QBOP) to set next rocks.

Tool 08

Issues List & IDS Process

IDS — Identify, Discuss, Solve — is how you actually fix problems instead of talking around them forever. The Issues List is the shared, visible list of everything getting in the way.

The IDS Rule

Whoever identifies an issue owns putting it on the list — not solving it alone. The team solves it together in the L10. Issues are prioritized by impact, not urgency. An issue that recurs 3x becomes a Rock to fix systemically. No "holy avoidance" — name things clearly.

Add issues in the Google Sheet — this list updates automatically
IssueAdded ByPriority

The 3-Step IDS Process

  • Identify — State the real issue, not the symptom. "Sundays feel rushed" → issue is "no documented run-of-show assigned." Be specific.
  • Discuss — Open the floor until the root issue is truly clear. Don't jump to solutions. Time-boxed to ~15 min/issue.
  • Solve — Create one To-Do: one owner, one due date. Issue is resolved or removed. Done.

LS-Specific IDS Culture Notes

  • Name personnel and pastoral issues clearly. Calling it "tension" doesn't solve it.
  • Issues belong to the team, not the person who raised them. No retaliation culture.
  • Never escalate to Ryan what Brad can solve. Never escalate to Brad what a director can solve.
  • IDS is not a venting session. Time-box discussion.
  • The Stepping Stones transition (post-Feb 2025) created systemic issues — these are Rocks, not just IDS items.
  • The Wonder gap means issues may not surface until they're critical. Build in explicit "What are we missing?" as a standing L10 prompt.
IDS to Rock Escalation Rule of 3
When an issue keeps coming back, it needs a Rock, not just a To-Do
  • Issue appears in IDS for the 1st time → To-Do with owner + due date
  • Issue reappears for 2nd time → escalate priority; examine root cause
  • Issue appears for 3rd time → becomes a company Rock next quarter
  • Issues that involve "Right Person / Right Seat" → always escalate to Brad for private 1-on-1 follow-up
  • Building / facility issues → Ryan + Brad own together; surface at every QBOP
Tool 09 — Admin

Google Sheets Setup & Live Data

This page can pull live data from a Google Sheet via GitHub. Follow these steps once to connect everything. After that, just update the Sheet — the page updates automatically.

How It Works

Google Sheet (you edit) → Apps Script (auto-runs) → GitHub JSON file → This dashboard (fetches on load). You enter data in the Sheet. The Script pushes it to a JSON file in your GitHub repo. This page reads that JSON every time the Staff panel opens.

Step 1 — Create the Google Sheet

Create a new Google Sheet named "LS South Reno Staff OS" with these tabs:

  1. Scorecard — Columns: Metric | Owner | Goal | Current | Status (green/red/yellow) | Notes
  2. Rocks — Columns: Title | Owner | Quarter | Status (on-track/off-track/complete) | Type (company/individual) | Notes
  3. Issues — Columns: Issue | AddedBy | Priority (high/med/low) | Status (open/resolved)
  4. VTO — Columns: Key | Value (for any V/TO fields you want to update live)

Step 2 — Create a GitHub Repository

  1. Create a new GitHub repo (e.g., lssr-staff-os) and enable GitHub Pages
  2. Add a folder called data/ and create a file data/eos-data.json with content: {}
  3. Generate a GitHub Personal Access Token (Settings → Developer Settings → Tokens) with repo scope
  4. Note your repo in the format: LSSouthReno/lssr-staff-os

Step 3 — Add Apps Script to the Google Sheet

In the Google Sheet, go to Extensions → Apps Script and paste this code:

// LS South Reno Staff OS — Google Apps Script // Paste into Extensions → Apps Script in your Google Sheet // Set GITHUB_TOKEN in Script Properties (Project Settings → Script Properties) const GITHUB_REPO = 'LSSouthReno/lssr-staff-os'; const GITHUB_FILE = 'data/eos-data.json'; function getSpreadsheet() { let ss = SpreadsheetApp.getActiveSpreadsheet(); if (!ss) { const id = PropertiesService.getScriptProperties().getProperty('SPREADSHEET_ID'); if (id) ss = SpreadsheetApp.openById(id); } if (!ss) throw new Error('Cannot find spreadsheet. Open this script from Extensions → Apps Script inside your sheet, or add SPREADSHEET_ID to Script Properties.'); return ss; } function pushToGitHub() { const ss = getSpreadsheet(); const data = { meta: { lastUpdated: new Date().toISOString().split('T')[0], quarter: getCell(ss, 'VTO', 'quarter') || 'Q2 2026' }, scorecard: getSheetData(ss, 'Scorecard', ['metric','owner','goal','current','status','notes']), rocks: getSheetData(ss, 'Rocks', ['title','owner','quarter','status','type','notes']), issues: getSheetData(ss, 'Issues', ['issue','addedBy','priority','status']) }; const token = PropertiesService.getScriptProperties().getProperty('GITHUB_TOKEN'); if (!token) { Logger.log('ERROR: Set GITHUB_TOKEN in Script Properties'); return; } const getUrl = `https://api.github.com/repos/${GITHUB_REPO}/contents/${GITHUB_FILE}`; const getRes = UrlFetchApp.fetch(getUrl, { headers: { 'Authorization': `token ${token}`, 'Accept': 'application/vnd.github.v3+json' }, muteHttpExceptions: true }); const sha = JSON.parse(getRes.getContentText()).sha || null; const body = { message: `Update EOS data — ${data.meta.lastUpdated}`, content: Utilities.base64Encode(JSON.stringify(data, null, 2)), sha: sha }; const putRes = UrlFetchApp.fetch(getUrl, { method: 'PUT', headers: { 'Authorization': `token ${token}`, 'Accept': 'application/vnd.github.v3+json', 'Content-Type': 'application/json' }, payload: JSON.stringify(body), muteHttpExceptions: true }); const result = JSON.parse(putRes.getContentText()); Logger.log(result.commit ? '✅ Pushed to GitHub: ' + result.commit.sha : '❌ Error: ' + JSON.stringify(result)); } function getSheetData(ss, sheetName, headers) { const sheet = ss.getSheetByName(sheetName); if (!sheet) { Logger.log('Sheet not found: ' + sheetName); return []; } const rows = sheet.getDataRange().getValues(); return rows.slice(1) .filter(r => r[0] && r[0].toString().trim() !== '') .map(r => { const obj = {}; headers.forEach((h, i) => { obj[h] = r[i] ? r[i].toString().trim() : ''; }); return obj; }); } function getCell(ss, sheetName, key) { const sheet = ss.getSheetByName(sheetName); if (!sheet) return null; const data = sheet.getDataRange().getValues(); for (const row of data) { if (row[0] && row[0].toString().toLowerCase() === key.toLowerCase()) return row[1].toString(); } return null; } function createTrigger() { ScriptApp.newTrigger('pushToGitHub').timeBased().everyHours(1).create(); Logger.log('Trigger created — will push every hour'); } function onOpen() { SpreadsheetApp.getUi().createMenu('🏛 Staff OS') .addItem('Push to GitHub Now', 'pushToGitHub') .addItem('Set Up Hourly Auto-Push', 'createTrigger') .addToUi(); }

Step 4 — Set Script Properties

  1. In Apps Script editor, click Project Settings (⚙) → Script Properties
  2. Add property: Key = GITHUB_TOKEN, Value = your GitHub personal access token
  3. The GITHUB_REPO constant is already set to LSSouthReno/lssr-staff-os
  4. Click Run → pushToGitHub to test. Check the Execution Log.

Step 5 — You're Live

The dashboard is already connected. The data URL is:

https://lssouthreno.github.io/lssr-staff-os/data/eos-data.json

Every time the Staff panel opens, it fetches the latest data. The green dot in the Snapshot tab confirms live data is loaded.
